## Service split (recommended)

- API service
- Web service
- Docs service (optional but recommended)
- Managed Postgres/Redis (or equivalent)

## Important repo-state note

Treat Render setup as **explicit service configuration** (Dockerfile + env per service).

- Do not assume turnkey blueprint automation unless your branch includes a verified Render blueprint file.
- Always verify file paths and env names against current repository state.

## Docker assets to reference

- API: `docker/builder/api/Dockerfile.api.prod`
- Web: `docker/builder/web/Dockerfile.web.build-time.prod`
- Docs: `docker/builder/doc/Dockerfile.doc.build-time.prod`

## Render-oriented sequence (safe)

1. Provision services and environment variables.
2. Configure build/start commands per service.
3. Configure inter-service URLs (`NEXT_PUBLIC_API_URL`, app/doc URLs, auth origins).
4. Deploy API first, then web.
5. Deploy docs service.
6. Run end-to-end smoke checks.

## Common pitfalls

- Misaligned public URL variables between API and web
- Missing auth callback/redirect URL updates
- Mismatch between declared Dockerfile path and actual repository structure
